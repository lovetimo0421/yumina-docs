Yumina Prompt Builder — Architecture Graph

Complete visualization of how the API assembles LLM prompts: from world entries to the final message array sent to the model.

1 — Entry System & Sections

Every world is composed of WorldEntry objects. Each entry has a section (where it goes in the prompt), a position (sort key within that section), and a role (semantic type). Entries are the atomic building blocks of all prompts.

Section alwaysSend Delivery Purpose
system-presets true Before chat history — stable prefix Core character, scenario, style, custom entries, official presets
examples true Parsed into user/assistant pairs Example dialogue (<START> / {{user}}: ... {{char}}: ...)
world-info false Keyword-triggered, after history Lorebook entries — injected only when keywords match recent messages
chat-history true Injected N msgs from end (depth) In-context reminders spliced into chat at specific depth
post-history true After all chat, before AI responds Jailbreak / final instructions — highest recency attention

Roles within entries: system, character, personality, scenario, lore, plot, style, example, greeting, custom. Greetings are used at session creation only. Examples are parsed into dialogue pairs.

2 — Official Presets

5 built-in presets auto-injected into system-presets. Users can override them via userPresetOverrides.

Fiction Mode pos 0

Content guidelines for mature/creative scenes. Sets the fiction framing.

Task pos 1

AI narrator role definition + consistency instructions.

Style pos 2

Character voice, complexity, imperfection guidance.

Instructions pos 3

Immersion, show-don't-tell, open-ended scene guidance.

CoT Bypass post-history

Prevents overthinking. Placed after all chat for maximum recency.

4 — Server-Side Pipeline (POST /api/sessions/:id/messages)
1

Load Context

  • Receive user message + optional attachments, model overrides
  • Load session → world definition + game state from DB
  • Check character creation flow (first turn special handling)
  • Apply user preset overrides to worldDef.entries
2

Initialize State

  • Create GameStateManager(worldDef, gameState)
  • Increment turn count
  • Session compaction check (summarize old messages if context is large)
  • Load non-compacted message history from DB
  • Populate macro context: lastMessage, lastUserMessage, lastCharMessage, model, etc.
3

Lorebook Retrieval

  • Scan last N messages (lorebookScanDepth, default 2)
  • retrieveLorebookEntries() — keyword matching against recent text
  • Supports: exact match, whole-word, fuzzy, secondary keywords (AND/NOT logic)
  • Recursive scanning: triggered entries' content can trigger more entries
  • Token budget cap prevents lorebook from consuming entire context
  • Output: { alwaysSend: WorldEntry[], triggered: WorldEntry[] }
4

Build Message Array PromptBuilder

This is the core assembly step. The PromptBuilder composes a ChatMessage[] array that becomes the LLM request. Each sub-step below adds messages to this array.

See the Final Message Array visualization below for the exact output structure.

5

4a. System Messages buildSystemMessages()

  • Collects all alwaysSend entries (excluding greetings, examples, chat-history, post-history)
  • Merges user prompts (converted to synthetic WorldEntry objects, position 10000+)
  • Sorts by position ascending
  • Injects active directives from rules at configured positions (top, before_char, after_char, auto, bottom)
  • System-role entries consolidated into one message; non-system entries flush and emit separately
  • All content interpolated via expandMacros()
6

4b. Example Dialogue buildExampleMessages()

  • Collects entries with role: "example"
  • Parses SillyTavern mes_example format: <START>\n{{user}}: ...\n{{char}}: ...
  • Emits [Example Chat] marker + alternating user/assistant message pairs
  • Each example block becomes a separate conversation example for the LLM
7

4c. Chat History + depth entries

  • [Start a new Chat] marker separates examples from real history
  • Real message history from DB (non-compacted messages only)
  • buildDepthEntries() — entries from chat-history section with depth values
  • Depth entries spliced into history at position history.length - depth from end
  • Example: depth=4 → entry injected 4 messages from the end of chat
8

4d. Lorebook + Format Block high-recency zone

  • buildLorebookMessages() — keyword-triggered entries wrapped in <world-context> tags
  • Placed AFTER history for maximum recency attention (LLMs attend more to recent content)
  • buildFormatBlock() — emits 4 sections:
    • <game-state> — variable summary grouped by category (stats/inventory/flags/etc.)
    • <behavior-rules> — per-variable AI behavior instructions
    • <directive-format> — how to emit [varId: op value] directives
    • <audio> — available tracks + [audio: trackId action] format
9

4e. Post-History buildPostHistoryEntries()

  • Entries from post-history section — placed LAST before AI responds
  • Highest possible recency attention — the LLM sees these immediately before generating
  • Typical use: jailbreak instructions, CoT bypass, final behavioral nudges
  • Can use any apiRole (system/user/assistant) — enables prefill tricks

After assembly, buildMessageHistory() applies token-budget trimming (oldest messages dropped first, most recent preserved). Then multimodal attachments are converted to provider format.

6 — Response Processing Loop

LLM Stream

SSE streaming via OpenRouterProvider. Text chunks sent to client in real-time.

→

Response Parser

Extracts [var: op value] directives + [audio: id action] from response text.

→

State + Rules

Apply effects → evaluate rules → apply rule effects → persist final state.

Response Parser Directive Extraction

Regex: /\[([\w\p{L}\p{N}.$-]+):\s*(set|add|subtract|...|\+|-|\*)?\s*("..."|\w+)?\]/gu

  • [health: -10] → subtract 10 from health
  • [gold: +50] → add 50 to gold
  • [location: set "forest"] → set location string
  • [hasKey: toggle] → toggle boolean
  • [stats: merge {"str": 5}] → merge JSON object
  • [audio: battle_bgm play] → trigger audio

Rules Engine Post-Response Evaluation

  1. Directives applied to state via GameStateManager.applyEffects()
  2. Rules evaluated with trigger context (turn count, keywords, state changes)
  3. 9 trigger types: state-change, variable-crossed, turn-count, session-start, keyword, ai-keyword, action, manual, every-turn
  4. 8 rule actions: modify-variable, inject-directive, remove-directive, send-context, toggle-entry, toggle-rule, notify-player, play-audio
  5. Rule effects applied → final state persisted to DB
7 — Rules → Next-Turn Prompt Injection

Rules that fire inject-directive or toggle-entry affect the next turn's prompt, not the current one. This creates a feedback loop between the AI's output and the world's state.

// Directive injection positions (used in buildSystemMessages): type DirectivePosition = "top" | "before_char" | "after_char" | "auto" | "bottom" // Example: Rule fires when health < 20 // → inject-directive: "The character is critically wounded. Describe pain and desperation." // → position: "after_char" (placed after character entries in system prompt) // → Next turn: this directive appears in buildSystemMessages() output // toggle-entry: Enables/disables world entries at runtime // → stored in gameState.ruleState.toggledEntries // → collectEntries() checks these overrides when building prompt

9 — Key Design Decisions

Prompt Caching Optimization

  • Static prefix: system-presets + examples → stable across turns → cacheable by LLM providers
  • Dynamic suffix: lorebook + format block + post-history → changes each turn → high recency attention
  • Cache breakpoint index passed to provider for explicit cache hints

Separation of Concerns

  • Engine (PromptBuilder): pure TypeScript, framework-agnostic, assembles messages
  • Server (messages.ts): orchestration — loads data, calls engine, streams to client, persists
  • App: displays streamed text + state changes, renders components
  • AI never executes code — all state changes go through the directive → parser → engine pipeline

Recency Attention Strategy

  • Lorebook entries placed AFTER chat history (not in system prompt) for better LLM attention
  • Format block (variables/directives) near the end — AI is most likely to follow recent instructions
  • Post-history entries are the very last thing before AI generates — maximum compliance
  • Depth entries allow surgical injection at specific points in chat history

Entry System Flexibility

  • 5 sections × 10 roles × arbitrary positions → extremely flexible prompt composition
  • User prompts (global presets) merge seamlessly as synthetic entries
  • Rules can inject/remove directives and toggle entries — dynamic prompt evolution
  • Session compaction summarizes old messages to stay within context limits
